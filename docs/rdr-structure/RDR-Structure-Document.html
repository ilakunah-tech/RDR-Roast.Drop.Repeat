1. Для онбординга

С чего начать: точка входа — App.kt (main() → RoastApp.start()). Загружается MainView.fxml, создаётся MainController. Вся основная логика записи обжарки и UI сходится в MainController.kt (крупный контроллер).

Где что лежит:

  • app — настройки (AppSettings, MachineConfig), создание источника данных (DataSourceFactory), запись (RoastRecorder, SamplingEngine, BbpService), API (ServerApi, BbpApi, ReferenceApi), темы и внешний вид.
  • ui — JavaFX: MainController + FXML-экраны (MainView, SettingsView, LoginView, PortsConfigView, AxesConfigView, EventButtonsView, RoastPropertiesView, FinishRoastView). Подпапки: ui.chart (графики JFreeChart), ui.control (слайдеры управления газ/воздух/барабан).
  • domain — модель обжарки: RoastProfile, TemperatureSample, события (EventType, RoastEvent, ControlEvent), кривые (CurveModel, StandardCurveModel, RorCurveModel), метрики (Phases, ChargeDetection, TurningPoint), BBP (Bbp, BbpAnalytics).
  • driver — подключение к оборудованию: интерфейс RoastDataSource (и RoastControl для управления), реализации: MODBUS (modbus.core, besca, diedrich), симулятор (SimulatorSource, AlogReplaySource), ErrorDataSource при ошибках.

Типичный сценарий: пользователь выбирает тип машины и подключается → MainController вызывает DataSourceFactory.create(config) → RoastRecorder.connect() → при Start начинается сбор сэмплов через SamplingEngine в RoastProfile → данные отображаются на CurveChartFx (JFreeChart).

2. Для рефакторинга

Границы модулей:

  • driver не должен зависеть от app/ui/domain (только от domain.TemperatureSample и своих интерфейсов). Соблюдается: app создаёт RoastDataSource через DataSourceFactory по MachineConfig.
  • domain — чистый: только данные и расчёты (кривые, метрики). Не зависит от app/driver/ui. Зависимости из app/ui к domain — норма.
  • app — оркестрация: настройки, конфиг, API, запись. Зависит от domain и driver. Сильная связка: RoastRecorder ↔ SamplingEngine ↔ BbpService.
  • ui — единственный потребитель app и части domain/driver для отображения. MainController — «жирный» класс: много ответственности (подключение, запись, графики, слайдеры, BBP, Reference, настройки). Кандидат на разбиение по сценариям или подконтроллерам.

Риски при изменениях:

  • Изменение RoastDataSource / MachineConfig затрагивает DataSourceFactory, все реализации driver и MainController.
  • Изменение RoastProfile / событий domain — RoastRecorder, BbpService, CurveChartFx, MainController.
  • Добавление нового типа машины: новый класс в driver + ветка в DataSourceFactory.create() + при необходимости настройки в AppSettings/SettingsController.

3. Документация продукта

Что делает приложение: RDR — десктопное приложение для записи профилей обжарки кофе. Подключается к ростеру (Besca, Diedrich, симулятор), снимает показания BT/ET по времени, позволяет отмечать события (Charge, First Crack, Drop и др.) и комментарии, при необходимости управляет газом/воздухом/барабаном (MODBUS). Поддерживается Between Batch Protocol (BBP), интеграция с сервером (ServerApi, BbpApi), референсные обжарки (ReferenceApi). Профили сохраняются локально (ProfileStorage), настраиваются порты и пресеты (SettingsManager, ConnectionPreset).

Где в коде:

  • Подключение к ростеру: DataSourceFactory, RoastRecorder.connect(), driver.*.
  • Запись кривой: RoastRecorder, SamplingEngine, RoastProfile.addSample().
  • Отображение графика: ChartPanelFx, CurveChartFx (JFreeChart), привязка к RoastProfile и CurveModel.
  • События и комментарии: RoastProfile (events, comments), ChartEventPopup, MainController.
  • Управление (слайдеры): RoastControl, ControlSliderPanel, ModbusCommandExecutor (write register/coil).
  • Настройки и пресеты: AppSettings, SettingsController, SettingsManager, MachineConfig.

Уровень 1 Пакеты и связи

Четыре основных пакета: app, ui, domain, driver. Направление зависимостей: ui → app, domain; app → domain, driver; domain независим.

  • com.rdr.roast — только App.kt (точка входа).
  • app — DataSourceFactory, RoastRecorder, SamplingEngine, BbpService, ProfileStorage, SettingsManager, AppSettings (в т.ч. MachineConfig), ServerApi, BbpApi, ReferenceApi, CatalogApi, ConnectionTester, RoasterDiscovery, ThemeSupport, AppearanceSupport и др.
  • ui — MainController; ui.chart: ChartPanelFx, CurveChartFx, ControlChartFx, ChartEventPopup; ui.control: ControlSliderPanel, DetachableSliderWindow; контроллеры настроек/портов/осей/событий/финиша/логина.
  • domain — Model.kt (RoastProfile, EventType, TemperatureSample, ControlEvent, RoastEvent, ProtocolComment, PhaseDuration и т.д.); domain.curves: CurveModel, AbstractCurveModel, StandardCurveModel, RorCurveModel, MovingAverageCurveModel; domain.metrics: Phases, ChargeDetection, TurningPoint; Bbp, BbpAnalytics.
  • driver — RoastDataSource, ConnectionState, RoastControl, ControlSpec, MachinePresets; modbus.core: ModbusConnectionCore, ModbusCommandExecutor, AbstractModbusRoasterSource; besca: BescaModbusSource, BescaModbusTcpSource; diedrich: DiedrichSource; simulator: SimulatorSource, AlogReplaySource; ErrorDataSource.

Диаграмма пакетов: см. раздел 9 (01-packages.puml / 01-packages.drawio).

Уровень 3 Зависимости между классами

Импорты и вызовы (ключевые):

  • MainController импортирует и использует: DataSourceFactory, RoastRecorder, ProfileStorage, ConnectionTester, MachineConfig, RoasterDiscovery, ReferenceApi, ServerApi, SettingsManager, EventQuantifiersConfig, domain (ControlEventType, EventType, RoastProfile, TemperatureSample, curves, metrics), driver (ConnectionState, RoastControl, ModbusCommandExecutor, ControlSpec), ui.chart (ChartPanelFx, CurveChartFx, ControlChartFx), ui.control (ControlSliderPanel, DetachableSliderWindow).
  • RoastRecorder создаёт и держит SamplingEngine и BbpService; при смене dataSource пересоздаёт их. Подписывается на dataSource.connectionState(), при старте записи вызывает samplingEngine.start { profile.addSample(it) }.
  • DataSourceFactory.create(MachineConfig) возвращает SimulatorSource | BescaModbusSource | BescaModbusTcpSource | DiedrichSource | ErrorDataSource (при отсутствии Phidget).
  • AbstractModbusRoasterSource реализует RoastDataSource и опционально RoastControl; использует ModbusConnectionCore для read/write; sampleFlow() в цикле читает регистры и эмитит TemperatureSample.
  • CurveChartFx получает RoastProfile и строит CurveModel (StandardCurveModel, RorCurveModel, MovingAverageCurveModel); подписывается на обновления и обновляет JFreeChart.
  • SettingsController использует AppSettings, ChartConfig, ConnectionPreset, MachineConfig, SettingsManager, ConnectionTester, ThemeSupport, AppearanceSupport и др.

Диаграмма классов: см. раздел 9 (02-classes-core.puml / 02-classes-core.drawio).

7. Три аспекта

7.1 Структура кода (импорты, вызовы, наследование)

Зависимости пакетов: ui → app, domain, driver; app → domain, driver; domain без зависимостей от app/ui/driver. Наследование: RoastDataSource реализуют AbstractModbusRoasterSource, BescaModbusSource, BescaModbusTcpSource, DiedrichSource, SimulatorSource, ErrorDataSource. RoastControl реализуют те же MODBUS-источники при поддержке управления. CurveModel реализуют StandardCurveModel, RorCurveModel, MovingAverageCurveModel (через AbstractCurveModel).

7.2 Потоки данных

Оборудование → RoastDataSource.sampleFlow() → SamplingEngine.collect() → RoastProfile.addSample() → StateFlow currentProfile → MainController подписан → CurveChartFx обновляет кривые и JFreeChart → отрисовка. Обратный поток управления: пользователь двигает слайдер → MainController (или ControlSliderPanel) → RoastControl.setControl() → ModbusCommandExecutor.writeSingle/writeCoil → устройство.

Диаграмма потока данных: см. раздел 9 (03-dataflow.puml / 03-dataflow.drawio).

7.3 Конфигурация и запуск

Последовательность: main() → RoastApp.start() → ThemeSupport, загрузка CSS → FXMLLoader.load(MainView.fxml) → создаётся MainController, инжект FXML → initialize(): загрузка SettingsManager/AppSettings, MachineConfig, DataSourceFactory.create(config) → RoastRecorder(dataSource), привязка stateFlow/currentProfile к UI, создание ChartPanelFx/CurveChartFx, загрузка списка профилей (ProfileStorage). При нажатии Connect: recorder.connect() → dataSource.connect() → connectionState обновляется → UI переходит в MONITORING.

Диаграмма запуска: см. раздел 9 (04-startup.puml / 04-startup.drawio).

9. Диаграммы (PlantUML и draw.io)

В этой папке лежат диаграммы в двух форматах. Для визуального просмотра:

  • PlantUML — откройте 01-packages.puml, 02-classes-core.puml, 03-dataflow.puml, 04-startup.puml в PlantUML online или в IDE с плагином PlantUML / экспортируйте в PNG/SVG.
  • draw.io — откройте 01-packages.drawio, 02-classes-core.drawio, 03-dataflow.drawio, 04-startup.drawio в draw.io (desktop или браузер).

Содержание диаграмм:

  • 01-packages — пакеты app, ui, domain, driver и связи между ними (уровень 1).
  • 02-classes-core — ключевые классы (RoastDataSource, RoastRecorder, MainController, ChartPanelFx, RoastProfile, CurveModel, реализации driver) и зависимости (уровень 2–3).
  • 03-dataflow — поток данных: User → MainController → RoastRecorder → SamplingEngine → RoastDataSource → RoastProfile → CurveChartFx → User; управление слайдерами.
  • 04-startup — последовательность запуска: JVM → App → JavaFX → MainView.fxml → MainController → SettingsManager, DataSourceFactory → RoastRecorder → RoastDataSource.
Как получить PDF: откройте этот файл (RDR-Structure-Document.html) в браузере и выберите «Печать» → «Сохранить как PDF» или «Microsoft Print to PDF». Все разделы и таблицы настроены для печати (разрывы страниц, фон). Диаграммы в виде картинок можно добавить в PDF, предварительно сгенерировав PNG из .puml (PlantUML) или экспортировав из draw.io и вставив в этот HTML перед печатью.
