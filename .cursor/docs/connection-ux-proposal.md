# Подключение к ростерам в RDR: UX и техническая реализация

Предложение по тому, как сделать подключение удобным для обычного пользователя и как это реализовать технически и в UI.

---

## 1. Цели для обычного пользователя

- **Минимум шагов:** выбрать «что подключаю» → при необходимости «как» (COM / сеть / USB) → один раз настроить и сохранить профиль.
- **Понятный статус:** всегда видно, подключён ли ростер и к чему (или причина ошибки).
- **Быстрое переключение:** смена машины/локации через выбор сохранённого профиля без копания в портах и регистрах.
- **Меньше ошибок:** по возможности выбирать порт из списка, проверять подключение до старта обжарки.

---

## 2. UI/UX предложения

### 2.1 Главный экран (MainView)

**Статус подключения рядом с кнопкой Start/Connect:**

- Всегда показывать короткую строку состояния, например:
  - `● Besca (COM4)` — подключено, устройство и порт;
  - `● Besca (10.0.0.9)` — подключено по TCP;
  - `○ Отключено` — нет подключения;
  - `⟳ Подключение...` — идёт подключение;
  - `⚠ Ошибка: порт занят` — ConnectionState.Error с кратким текстом.
- Размещение: в нижней панели рядом с кнопками (слева от «Start» / «Connect») или в верхней панели справа от «Настройки».
- Цвет: зелёный — подключено, серый — отключено, оранжевый/красный — ошибка.

Так пользователь сразу видит, нужно ли нажимать «Connect» и есть ли проблема.

### 2.2 Настройки → вкладка «Подключение»

Текущая логика (источник → профиль → поля порта) уже хорошая. Улучшения:

1. **Порядок и группировка**
   - Блок 1: **«Источник данных»** — Simulator / Besca / Diedrich (как сейчас).
   - Блок 2: **«Профиль»** — выпадающий список сохранённых пресетов + «Загрузить» / «Сохранить как…» (как сейчас).
   - Блок 3: **«Параметры подключения»** — показывать только релевантные поля в зависимости от источника и способа (Serial / TCP / Phidget), без лишних полей.

2. **Упрощение выбора порта (Serial)**
   - Вместо только текстового поля COM-порта — **выпадающий список** с обнаруженными портами (COM1, COM3, COM4, …).
   - Опция «Другой…» или ручной ввод, если порта нет в списке (виртуальные COM, WSL и т.п.).
   - Список обновлять при открытии настроек и по кнопке «Обновить порты».

3. **Кнопка «Проверить подключение»**
   - В блоке параметров подключения кнопка «Проверить».
   - По нажатию: с текущими полями (без сохранения) создаётся временный `RoastDataSource`, вызывается `connect()`, через 2–5 сек проверяется `ConnectionState`.
   - Результат: тост или диалог «Подключено» / «Ошибка: &lt;сообщение&gt;», затем disconnect.
   - Пользователь может проверить настройки до сохранения и без выхода из настроек.

4. **Тексты и подсказки**
   - Короткие подсказки (tooltip) у полей: «Slave ID обычно 1», «Порт Modbus TCP по умолчанию 502».
   - Для Diedrich/Phidget — явно писать «Канал ET» / «Канал BT», как сейчас по сути и есть.

### 2.3 Поведение при сохранении настроек

- Оставить текущую логику: при «Сохранить» в настройках приложение применяет новый `machineConfig`, делает `disconnect()` → `dataSource = DataSourceFactory.create(config)` → `connect()`.
- Дополнительно: если новое подключение переходит в `Error`, показать краткое уведомление с текстом ошибки и оставить кнопку «Connect» на главном экране для повтора.

## 3. Техническая реализация

### 3.1 Что уже есть и не менять

- `RoastDataSource`, `ConnectionState` (Disconnected / Connecting / Connected / Error).
- `DataSourceFactory.create(MachineConfig)`.
- `MachineConfig` (machineType, transport, host, tcpPort, port, baudRate, slaveId, …).
- `ConnectionPreset` и `SettingsManager.loadPresets()` / `savePresets()`.
- В главном окне: при DISCONNECTED кнопка «Connect», при сохранении настроек — пересоздание источника и reconnect.

Этого достаточно, чтобы строить UX поверх.

### 3.2 Показ статуса на главном экране

- В `MainController` подписаться не только на `recorder.stateFlow`, но и на `recorder.dataSource.connectionState()`.
- По `ConnectionState` обновлять:
  - Текст и стиль виджета статуса (Label или маленькая панель).
  - Для `Error` — показывать `state.message` (обрезать по длине, если нужно).
- Новый элемент в `MainView.fxml`: например `lblConnectionStatus` в нижней панели или в top HBox.

### 3.3 Обнаружение COM-портов

- Использовать jSerialComm (уже в проекте для Modbus RTU): `SerialPort.getCommPorts()` и взять имена портов.
- В `SettingsController` при открытии вкладки «Подключение» (или по кнопке «Обновить») заполнять ComboBox списком портов.
- Сохранённое значение `machineConfig.port` подставлять в выбор; если такого порта нет в списке — показывать в комбобоксе как «COM4 (текущий)» или оставить поле ввода для произвольного имени.

### 3.4 Проверка подключения из настроек

- Вынести создание источника в фабрику (уже есть). Добавить утилиту или метод, например:
  - `ConnectionTester.test(config: MachineConfig): Result<ConnectionState, String>`:
    - Создать `DataSourceFactory.create(config)`, вызвать `connect()`, в корутине с таймаутом (3–5 сек) ждать перехода в `Connected` или `Error`.
    - Вернуть успех или текст ошибки, затем вызвать `disconnect()`.
- В `SettingsController` по «Проверить подключение» вызвать `ConnectionTester.test(configFromFields)` в фоне (корутины/Executor), результат показать в UI (Alert или тост). Не сохранять настройки и не менять глобальный `recorder.dataSource`.

### 3.5 Расширение в будущем

- Новые типы машин: добавить `MachineType`, ветку в `DataSourceFactory` и опцию в ComboBox «Источник данных».
- Новый транспорт: добавить `Transport`, поля в `MachineConfig` и условное отображение полей в настройках.
- Сохранение пресетов и привязка к «последнему использованному» профилю при следующем запуске уже поддерживаются через `SettingsManager.load()` и пресеты.

## 4. План внедрения по шагам

| Шаг | Что сделать | Сложность |
|-----|-------------|-----------|
| 1 | Добавить на главный экран виджет статуса (Label), подписать на `connectionState()`, показывать Connected/Disconnected/Connecting/Error | Низкая |
| 2 | В настройках: ComboBox для COM-порта с заполнением из jSerialComm, кнопка «Обновить порты» | Низкая |
| 3 | Кнопка «Проверить подключение» в настройках + `ConnectionTester` с таймаутом и отображением результата | Средняя |
| 4 | Улучшить тексты/подсказки и, при необходимости, группировку блоков во вкладке «Подключение» | Низкая |

Сначала имеет смысл сделать шаги 1 и 2 — они сразу делают подключение понятнее и уменьшают опечатки в имени порта. Шаг 3 сильно повышает уверенность пользователя перед сохранением. Шаг 4 — полировка.

---

## 5. Краткое резюме

- **Для пользователя:** один раз настроил профиль (с выбором порта из списка и при желании проверкой) → на главном экране всегда видно «подключено / отключено / ошибка» → переключение между ростерами через выбор пресета.
- **Технически:** оставляем текущую архитектуру (DataSource, Factory, MachineConfig, presets), добавляем отображение `connectionState` на главном экране, обнаружение COM-портов в настройках и опциональный `ConnectionTester` для кнопки «Проверить подключение».

Если нужно, можно начать с реализации шага 1 (статус на главном экране) и шага 2 (список COM-портов в настройках) в коде.
